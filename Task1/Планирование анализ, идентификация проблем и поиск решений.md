# Планирование: анализ, идентификация проблем и поиск решений

## 1. Анализ текущей системы

Система состоит из трёх основных приложений:

- **Онлайн-магазин** (Vue + Java Spring Boot)
- **CRM** (Vue + Java Spring Boot)
- **MES** (React + C#)

Интеграция между CRM и MES осуществляется через RabbitMQ.  
MES выполняет расчёт стоимости изделия. Расчёт занимает от 2–3 минут до 30 минут.

### Инфраструктура

- Каждый сервис развернут в одном инстансе.
- Каждая база данных имеет один инстанс (read/write).
- Используется облачная инфраструктура (EC2 + Managed DB).
- CI/CD частично ручной.
- E2E тестирование выполняется вручную.

Компания испытывает постоянный рост нагрузки (каждый месяц +100 заказов).  
После открытия B2B API поток заказов резко увеличился.

Производственные мощности достаточны, следовательно, основная проблема находится в ИТ-контуре.

---

## 2. Идентификация существующих и потенциальных проблем

### 2.1 Проблемы масштабируемости

1. Все приложения работают в одном инстансе — отсутствует горизонтальное масштабирование.
2. Базы данных имеют один инстанс — возможный bottleneck при росте чтений и записей.
3. Расчёт стоимости в MES выполняется долго и не масштабируется.
4. После открытия API резко увеличился поток заказов без изоляции нагрузки.

---

### 2.2 Проблемы интеграции и надёжности

5. Интеграция через RabbitMQ не описывает механизмы:
   - гарантированной доставки,
   - идемпотентности,
   - DLQ,
   - мониторинга очередей.
6. Возможна потеря сообщений при сбоях.
7. Возможна повторная обработка сообщений.
8. Нет прозрачной трассировки заказа между системами.

---

### 2.3 Проблемы производительности MES

9. Первая страница MES загружается медленно.
10. Фильтрация и пагинация не решают проблему.
11. Вероятно выполняются тяжёлые SQL-запросы по большой таблице заказов.
12. Нет специализированной read-модели для дашборда.

---

### 2.4 Проблемы процессов разработки

13. Ручные деплои в release и prod.
14. Ручное E2E тестирование.
15. Частые задержки релизов.
16. Отсутствие автоматизированной регрессии.

---

### 2.5 Потенциальные риски

17. Дальнейший рост нагрузки приведёт к деградации системы.
18. Потеря контрактов из-за нестабильности.
19. Потеря доверия B2B партнёров.
20. Перегрузка операторов и продавцов.

---

## 3. Инициативы по устранению проблем

### Инициатива 1. Масштабирование backend-сервисов

- Развернуть минимум 2 инстанса для каждого backend.
- Настроить балансировщик нагрузки.

---

### Инициатива 2. Выделение расчёта стоимости в отдельные воркеры

- Перевести расчёт в асинхронную модель.
- Добавить масштабируемые воркеры.
- Масштабировать по длине очереди.

---

### Инициатива 3. Внедрение Outbox/Inbox + DLQ

- Реализовать transactional outbox.
- Добавить идемпотентность обработки.
- Настроить DLQ и мониторинг.

---

### Инициатива 4. Оптимизация дашборда MES

- Добавить индексы.
- Внедрить keyset pagination.
- Создать read-модель для отображения заказов.

---

### Инициатива 5. Ограничение нагрузки от B2B API

- Ввести rate limiting.
- Ввести квоты для партнёров.
- Приоритизация B2C заказов.

---

### Инициатива 6. Внедрение мониторинга и наблюдаемости

- Метрики RabbitMQ.
- Метрики обработки заказов.
- Логирование с correlation-id.
- Настройка алертов.

---

### Инициатива 7. Автоматизация тестирования

- Автоматизация E2E сценариев.
- Интеграция автотестов в CI.
- Снижение ручной проверки.

---

### Инициатива 8. Разделение read/write нагрузки БД

- Внедрение read-replica.
- Использование кэширования для частых запросов.

---

## 4. Приоритизация инициатив

Приоритизация выполнена по следующим критериям:

- Влияние на бизнес.
- Снижение риска потери заказов.
- Повышение устойчивости системы.
- Соотношение эффект/затраты.

---

### Приоритет 1 (критически важные)

1. Выделение расчёта стоимости в воркеры  
2. Внедрение Outbox/Inbox + DLQ  
3. Оптимизация дашборда MES  

Обоснование:  
Эти инициативы напрямую влияют на зависание заказов и жалобы клиентов.

---

### Приоритет 2 (высокий)

4. Масштабирование backend-инстансов  
5. Ограничение нагрузки от B2B API  
6. Внедрение мониторинга  

Обоснование:  
Повышают устойчивость и управляемость системы.

---

### Приоритет 3 (средний)

7. Автоматизация тестирования  
8. Разделение read/write нагрузки БД  

Обоснование:  
Повышают зрелость системы, но не устраняют текущий кризис напрямую.

---

## 5. Целевая архитектура через полгода

Через полгода система должна обладать следующими характеристиками:

- Горизонтально масштабируемые backend-сервисы.
- Асинхронный расчёт стоимости через очередь и воркеры.
- Надёжная доставка событий (Outbox/Inbox).
- Быстрый дашборд MES через read-модель.
- Ограничение нагрузки от B2B API.
- Полноценный мониторинг и алерты.
- Частичная автоматизация тестирования.

Система должна быть устойчива к дальнейшему линейному росту заказов.

---

## 6. Если можно выполнить только три инициативы

Я бы выбрал:

1. Выделение расчёта стоимости в масштабируемые воркеры.  
2. Внедрение Outbox/Inbox + DLQ.  
3. Оптимизацию дашборда MES.  

Обоснование:

Расчёт стоимости — самое тяжёлое место системы.  
Надёжность интеграции — причина “потерянных” заказов.  
Медленный дашборд — замедляет производство и демотивирует операторов.

Эти три инициативы:

- стабилизируют поток заказов,
- устранят основные жалобы,
- предотвратят потерю клиентов,
- создадут основу для дальнейшего масштабирования.
