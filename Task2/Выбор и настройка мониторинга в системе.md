# Выбор и настройка мониторинга в системе

## 1. Введение

Компания «Александрит» сталкивается с ростом нагрузки, увеличением числа заказов и жалобами клиентов (B2C и API-партнёров). Заказы теряются, сроки производства нарушаются, система MES работает медленно, релизы задерживаются.

На текущий момент система подключена только к Яндекс.Метрике, что позволяет анализировать поведение пользователей онлайн-магазина, но не даёт полной картины работы распределённой системы (Shop → CRM → MES → RabbitMQ → DB).

Для стабилизации работы системы и дальнейшего масштабирования необходимо внедрить полноценный технический мониторинг.

---

# 2. Мотивация

## 2.1 Почему мониторинг необходим

Без мониторинга компания:

- не понимает, где теряются заказы;
- не видит, где возникает задержка обработки;
- не может прогнозировать рост нагрузки;
- не может гарантировать SLA API-партнёрам;
- теряет контракты и репутацию.

## 2.2 Что даст внедрение мониторинга бизнесу

1. Снижение количества потерянных заказов
2. Уменьшение времени реакции на инциденты
3. Возможность масштабирования инфраструктуры на основе данных
4. Повышение прозрачности процессов для руководства
5. Возможность формализации SLA для B2B API
6. Снижение количества жалоб клиентов

Мониторинг — это не техническая оптимизация, а инструмент защиты выручки.

---

# 3. Выбор подхода к мониторингу

В системе используются разные типы компонентов, поэтому применяется комбинированный подход.

## 3.1 RED (Rate, Errors, Duration)

Применяется для:

- Shop API
- CRM API
- MES API
- RabbitMQ (как сервис обработки сообщений)

Позволяет отслеживать:

- нагрузку
- ошибки
- задержки

## 3.2 USE (Utilization, Saturation, Errors)

Применяется для:

- серверов приложений
- баз данных
- инфраструктуры

Позволяет выявлять:

- нехватку ресурсов
- saturation
- ошибки инфраструктуры

# 5. Дополнительные бизнес-метрики

## 5.1 Время прохождения заказа по статусам

Измерять время между статусами:

- SUBMITTED → PRICE_CALCULATED
- PRICE_CALCULATED → MANUFACTURING_STARTED
- MANUFACTURING_STARTED → MANUFACTURING_COMPLETED
- COMPLETED → SHIPPED

**Labels:**

- channel (B2C / API)
- partner_id

Позволяет выявлять узкие места.

---

## 5.2 Количество заказов в каждом статусе

Позволяет определить зависание заказов.

Пример:  
Если много заказов в SUBMITTED и мало в PRICE_CALCULATED — проблема в MES.

---

# 6. Технологический стек мониторинга

Предлагаемый стек:

- Prometheus — сбор метрик
- Grafana — визуализация
- Alertmanager — алерты
- Loki — централизованные логи
- OpenTelemetry — трассировка запросов
- Sentry — мониторинг ошибок

Это позволит:

- отслеживать путь одного заказа через всю систему;
- быстро находить место сбоя;
- анализировать деградацию производительности.

---

# 7. Ожидаемые результаты после внедрения

После внедрения мониторинга компания сможет:

- выявить причину потери заказов;
- обнаружить узкое место в расчёте стоимости;
- определить необходимость масштабирования MES;
- контролировать API-нагрузку партнёров;
- сократить время реакции на инциденты;
- повысить стабильность и доверие клиентов.

---

# 8. Вывод

Основная проблема «Александрита» — отсутствие наблюдаемости в условиях роста нагрузки.

Компания масштабировала бизнес (открыла API), не внедрив технический контроль состояния системы.

Мониторинг — необходимое условие для:

- стабилизации текущей архитектуры;
- масштабирования;
- сохранения контрактов;
- предотвращения потери заказов.

## Внедрение мониторинга является приоритетной инициативой для обеспечения устойчивого роста компании.

# 9. План действий

Ниже представлен высокоуровневый план внедрения мониторинга. Данный раздел можно рассматривать как драфт технического задания.

## 9.1 Подготовка инфраструктуры мониторинга

1. Создать отдельный инстанс time-series базы данных (Prometheus).
2. Развернуть Grafana для визуализации метрик.
3. Настроить Alertmanager для обработки алертов.
4. Развернуть Loki для централизованного сбора логов.
5. Подключить OpenTelemetry для распределённого трейсинга.
6. Подключить Sentry для отслеживания ошибок приложений.

---

## 9.2 Инструментирование приложений

1. Добавить экспорт метрик (Spring Boot Actuator + Micrometer) в:
   - Shop API
   - CRM API

2. Добавить экспорт метрик в MES (C# + Prometheus exporter).

3. Настроить сбор:
   - HTTP-метрик (RPS, latency, status codes)
   - метрик бизнес-уровня (статусы заказов)
   - метрик расчёта стоимости (время выполнения операции)

---

## 9.3 Настройка мониторинга RabbitMQ

1. Подключить RabbitMQ Exporter.
2. Настроить сбор:
   - backlog очередей
   - dead-letter сообщений
   - времени доставки сообщений

---

## 9.4 Мониторинг баз данных

1. Подключить DB Exporter.
2. Настроить сбор:
   - количества соединений
   - использования памяти
   - размера базы
   - времени выполнения запросов

---

## 9.5 Настройка дашбордов

Создать в Grafana следующие дашборды:

1. Обзор системы (System Overview)
2. API-производительность (Shop / CRM / MES)
3. RabbitMQ Dashboard
4. Database Dashboard
5. Бизнес-метрики (жизненный цикл заказа)

---

## 9.6 Настройка алертов

Настроить алерты для:

- роста HTTP 500 выше порога
- превышения latency p95
- накопления backlog в RabbitMQ
- превышения CPU / Memory
- превышения числа DB connections

---

# 10. Показатели насыщенности (Saturation Thresholds)

Насыщенность показывает, что система приближается к пределу своих возможностей.

Ниже определены ключевые пороги.

---

## 10.1 CPU

Порог:

- Warning: 70% более 5 минут
- Critical: 85% более 5 минут

Почему:  
После 80–85% начинается деградация производительности и рост latency.

Действия:

- автоматически добавить инстанс (если включён autoscaling)
- создать тикет в Jira
- уведомить DevOps через Alertmanager

---

## 10.2 Memory

Порог:

- Warning: 75%
- Critical: 90%

Почему:  
При приближении к 100% возможен OOM и падение приложения.

Действия:

- уведомление DevOps
- проверка memory leak
- масштабирование

---

## 10.3 RabbitMQ Backlog

Порог:

- Warning: > 1000 сообщений в очереди более 10 минут
- Critical: > 5000 сообщений

Почему:  
Backlog означает, что обработчики не успевают потреблять сообщения, возможна потеря SLA.

Действия:

- увеличить количество consumer-инстансов
- проверить падения обработчиков
- уведомить команду

---

## 10.4 Dead-letter messages

Порог:

- Любое появление > 0 — Warning
- > 10 в течение 5 минут — Critical

Почему:  
Dead-letter = потенциально потерянный заказ.

Действия:

- автоматическое создание инцидента
- уведомление команды
- при необходимости — временная остановка приёма заказов через API

---

## 10.5 Latency (p95)

Порог:

- Shop API: > 2 секунды
- CRM API: > 2 секунды
- MES price calculation: > 5 минут

Почему:  
MES расчёт может занимать минуты, но рост выше 5 минут говорит о деградации.

Действия:

- проверка нагрузки
- масштабирование
- временное ограничение API-партнёров

---

## 10.6 DB Connections

Порог:

- Warning: 70% от max_connections
- Critical: 90%

Почему:  
Достижение лимита приведёт к отказу новых запросов.

Действия:

- масштабирование базы
- внедрение connection pooling
- оптимизация запросов

---

# 11. Реакция системы при превышении порогов

При превышении критических порогов должны автоматически выполняться:

1. Отправка уведомления в Slack / Telegram.
2. Создание тикета в системе управления задачами.
3. Уведомление ответственного инженера.
4. При необходимости — запуск автоскейлинга.
5. Для бизнес-критичных инцидентов — уведомление продакт-менеджера.

---

# 12. Итог

Внедрение мониторинга позволит:

- обеспечить наблюдаемость распределённой системы;
- предотвратить потерю заказов;
- гарантировать SLA API-партнёрам;
- обеспечить устойчивое масштабирование;
- снизить количество инцидентов и жалоб клиентов.

Мониторинг является обязательным шагом для стабилизации архитектуры «Александрита» и дальнейшего роста бизнеса.
