# Архитектурное решение по трейсингу

---

## 1. Анализ системы в контексте трейсинга

В текущей архитектуре отсутствует сквозная наблюдаемость прохождения заказа между сервисами. Заказ проходит через несколько распределённых компонентов (Shop API, CRM API, MES API, RabbitMQ, базы данных, S3), при этом нет механизма, позволяющего увидеть полный жизненный цикл одного заказа.

Это приводит к следующим проблемам:

- Невозможно определить, где именно «завис» заказ.
- Сообщения в RabbitMQ могут теряться или обрабатываться повторно.
- При ошибке расчёта стоимости нет прозрачности.
- Заказы могут находиться в неконсистентных статусах между системами.
- Служба поддержки вынуждена искать информацию вручную в логах.

---

## 2. Компоненты, которые необходимо покрыть трейсингом

Трейсинг должен покрывать все backend-компоненты, участвующие в жизненном цикле заказа:

- Shop API (Spring Boot)
- CRM API (Spring Boot)
- MES API (C#)
- RabbitMQ
- Интеграция с S3 (хранение 3D-файлов)
- Shop DB (PostgreSQL)
- MES DB (PostgreSQL)
- API для внешних B2B клиентов

Frontend-приложения (Vue, React) не являются приоритетом для трейсинга, так как ключевая проблема связана с межсервисным взаимодействием.

---

## 3. Точки потенциальных сбоев

### 3.1 Shop API

- Ошибка записи в БД
- Потеря запроса при высокой нагрузке
- Таймаут при загрузке 3D-файла
- Ошибка передачи данных в CRM

### 3.2 CRM API

- Ошибка при получении сообщения из RabbitMQ
- Неконсистентное изменение статуса
- Ошибка публикации события

### 3.3 RabbitMQ

- Потеря сообщений
- Попадание сообщений в DLQ
- Повторная обработка
- Отсутствие подтверждения доставки

### 3.4 MES API

- Долгий расчёт стоимости (до 30 минут)
- Блокировки в БД
- Ошибки при смене статусов
- Неполная обработка сообщения

### 3.5 MES Dashboard

- Медленные запросы к БД
- Заказ существует, но не отображается оператору

## 4. Данные, которые должны попадать в трейсинг

Для обеспечения сквозной наблюдаемости в систему трейсинга должны передаваться следующие данные:

### Идентификаторы

- `trace_id`
- `span_id`
- `parent_span_id`
- `order_id`
- `user_id` или `api_client_id`

### HTTP-данные

- HTTP method
- endpoint
- response_status
- latency
- source_system
- target_system

### RabbitMQ

- message_id
- routing_key
- queue_name
- retry_count

### Бизнес-данные

- order_status_before
- order_status_after
- file_size
- polygon_count (если доступно)

### Ошибки

- exception_type
- exception_message

## 5. Мотивация

### Техническая мотивация

Внедрение трейсинга позволит:

- Снизить MTTR (Mean Time To Recovery)
- Ускорить поиск причин инцидентов
- Обнаруживать потерянные сообщения
- Видеть реальные задержки в цепочке обработки заказа
- Обнаруживать узкие места в системе

### Бизнес-метрики, на которые повлияет решение

- Снижение количества просроченных заказов
- Снижение числа жалоб B2C и B2B клиентов
- Снижение потери контрактов
- Повышение доверия к API
- Улучшение SLA выполнения заказов

---

## 6. Предлагаемое решение

### 6.1 Технологический стек

- OpenTelemetry (SDK для Java и .NET)
- OpenTelemetry Collector
- Jaeger (или Grafana Tempo) для хранения и визуализации трейсов
- Grafana для визуализации
- Prometheus для метрик (в рамках расширения решения)

---

### 6.2 Инструментирование сервисов

#### Shop API и CRM API (Spring Boot)

- Подключение OpenTelemetry Java SDK
- Автоматическая HTTP-инструментация
- Инструментация JDBC
- Инструментация RabbitMQ

#### MES API (C#)

- OpenTelemetry .NET SDK
- ASP.NET instrumentation
- PostgreSQL instrumentation
- RabbitMQ instrumentation

---

### 6.3 Прокидывание контекста

- Использовать стандарт W3C Trace Context
- Передавать `trace_id` через HTTP-заголовки
- Передавать `trace_id` в headers сообщений RabbitMQ
- Обеспечить сохранение correlation id между сервисами

---

### 6.4 Архитектура решения

- OpenTelemetry SDK в каждом backend-сервисе
- OpenTelemetry Collector
- Jaeger (или Tempo)

## 7. Дополнительное задание: автоматический контроль прохождения заказа

### 7.1 Идея

На основе данных трейсинга реализуется автоматический мониторинг жизненного цикла заказа.

Система должна:

- Отслеживать длительность нахождения заказа в каждом статусе
- Обнаруживать оборванные трейсы
- Контролировать SLA расчёта стоимости и производства

---

### 7.2 Правила алертинга (примеры)

- Заказ в статусе SUBMITTED более 35 минут без PRICE_CALCULATED
- Нет MANUFACTURING_STARTED более 48 часов
- Сообщение находится в очереди более N минут
- Трейс оборвался и не завершился

---

### 7.3 Компоненты алертинга

- Prometheus
- Alertmanager
- Grafana

## 8. Компромиссы

- Увеличение нагрузки на систему (~5–10%)
- Рост затрат на хранение данных
- Возможная сложность доработки MES (C#)
- Длинные трейсы при длительных операциях расчёта стоимости
- Необходимость обучения команды

---

## 9. Аспекты безопасности

- Доступ к системе трейсинга только через VPN
- Интеграция с корпоративным SSO
- RBAC (роль «Поддержка», «DevOps», «Разработчик»)
- Шифрование трафика (TLS)
- Ограничение срока хранения трейсов
- Маскирование персональных данных (PII)

---

# Итог

Внедрение распределённого трейсинга обеспечит сквозную прозрачность жизненного цикла заказа, сократит время диагностики инцидентов, снизит потери заказов и повысит устойчивость системы при росте нагрузки и расширении B2B-направления.
