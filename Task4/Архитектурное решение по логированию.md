# Архитектурное решение по логированию

## 1. Анализ текущей ситуации

В системе компании «Александрит» отсутствует централизованное логирование.  
Ошибки и нестандартные ситуации разбираются со слов клиентов и операторов.

Это приводит к:

- Долгому поиску причины инцидентов
- Потере заказов
- Росту количества просроченных заказов
- Потере B2B-контрактов
- Перегрузке команды поддержки
- Задержкам релизов

Система состоит из:

- Online Shop (Vue + Spring Boot)
- CRM (Vue + Spring Boot)
- MES (React + C#)
- RabbitMQ
- Базы данных (Managed Yandex Cloud)
- API для B2B клиентов

Заказ проходит через несколько систем, поэтому логирование должно быть централизованным и коррелируемым.

# 2. Какие логи необходимо собирать

## 2.1 Логи бизнес-событий (уровень INFO)

### Изменение статуса заказа

Логируем:

- timestamp
- order_id
- previous_status
- new_status
- source_system
- user_id (если применимо)
- correlation_id

---

### Создание заказа

- timestamp
- order_id
- customer_id
- channel (B2C / B2B API)
- correlation_id

---

### Расчёт стоимости в MES

- timestamp
- order_id
- polygons_count
- calculation_time
- result
- correlation_id

---

### Получение и обработка сообщения из RabbitMQ

- timestamp
- queue_name
- message_id
- order_id
- processing_result
- retry_count
- correlation_id

---

### Производственные этапы

- MANUFACTURING_STARTED
- MANUFACTURING_COMPLETED
- PACKAGING
- SHIPPED

Логируем:

- timestamp
- order_id
- operator_id
- duration
- correlation_id

---

## 2.2 Технические логи

### HTTP-запросы

- method
- endpoint
- status_code
- latency
- user_id
- ip
- correlation_id

---

### Ошибки (уровень ERROR)

- exception_type
- message
- stacktrace
- order_id (если есть)
- correlation_id

---

## 2.3 Инфраструктурные логи

- RabbitMQ (dead-letter, reject, nack)
- Slow query log БД
- Web server access logs
- Логи контейнеров/EC2

---

# 3. Уровни логирования

| Уровень | Использование                                 |
| ------- | --------------------------------------------- |
| INFO    | Бизнес-события, успешные операции             |
| WARN    | Retry, долгие операции, повторная обработка   |
| ERROR   | Исключения, потеря соединения, сбой обработки |
| DEBUG   | Только dev или временно в prod                |

---

# 4. Мотивация внедрения логирования

## Бизнес-эффект

- Снижение количества потерянных заказов
- Снижение времени расследования инцидентов (MTTR)
- Уменьшение нагрузки на поддержку
- Сохранение B2B-контрактов
- Повышение доверия клиентов

## Технические метрики, которые улучшатся

- MTTR (Mean Time To Recovery)
- Количество потерянных сообщений
- Количество HTTP 500
- Время нахождения заказа в статусе
- Время реакции на инцидент

# 6. Предлагаемое решение

## 6.1 Архитектура

Внедряется централизованная система сбора логов:

- Логирование в JSON-формате во всех backend-сервисах
- FluentBit (или Filebeat) на каждом инстансе
- OpenSearch как централизованное хранилище
- OpenSearch Dashboards для анализа
- Alerting внутри OpenSearch

## 6.2 Технологический выбор

Выбирается OpenSearch по следующим причинам:

- Лицензия Apache 2.0
- Отсутствие vendor lock-in
- Поддержка alerting
- Поддержка RBAC
- Возможность масштабирования
- Отсутствие высокой стоимости владения (в отличие от Splunk)

# 7. Политика безопасности

- Доступ только через VPN
- RBAC (разделение доступа DevOps / Backend / Support)
- Маскирование персональных данных (email, телефон)
- Запрет логирования платёжных данных
- Разделение индексов по системам

# 9. Превращение логирования в систему анализа

## 9.1 Алертинг

Настраиваются алерты:

- > 5% HTTP 500
- Очередь RabbitMQ > N сообщений
- Заказ в одном статусе > X часов
- Всплеск создания заказов (> 10 000 в секунду)
- Рост ERROR-логов

---

## 9.2 Поиск аномалий

- Резкий рост количества заказов
- Резкое падение количества заказов
- Увеличение времени расчёта стоимости
- Повторная обработка сообщений

---

# 10. Критерии выбора технологии (дополнительное задание)

| Критерий         | ELK             | OpenSearch | Splunk        |
| ---------------- | --------------- | ---------- | ------------- |
| Лицензия         | Elastic License | Apache 2.0 | Проприетарная |
| Стоимость        | Средняя         | Низкая     | Высокая       |
| Масштабируемость | Высокая         | Высокая    | Высокая       |
| Alerting         | Есть            | Есть       | Есть          |
| RBAC             | Есть            | Есть       | Есть          |
| Vendor lock-in   | Частично        | Нет        | Да            |
| Поддержка облака | Да              | Да         | Да            |

Выбор: **OpenSearch**, как оптимальный баланс стоимости, гибкости и функциональности.

---

# 11. Ожидаемый результат через 3–6 месяцев

- Время расследования инцидентов снижено минимум в 3 раза
- Потерянные заказы практически исключены
- Поддержка работает через дашборды, а не через рассказы клиентов
- Появилась прозрачность прохождения заказа по всем системам
- Компания готова к дальнейшему росту нагрузки
