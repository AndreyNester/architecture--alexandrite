# Архитектурное решение по кешированию (Task5)

> Документ описывает подход к кешированию в MES для ускорения работы операторов и снижения нагрузки на БД.

---

## 1. Контекст и проблема

MES используется операторами для работы с заказами. На первой странице MES отображается дашборд со списком заказов в работе по статусам (с фильтром и пагинацией). С ростом числа заказов страница стала прогружаться медленно, и пользователи жалуются на скорость работы.

Дополнительно нагрузка на MES возросла после открытия публичного API для внешних партнёров: заказов стало значительно больше, а SLA и ожидания по скорости работы — выше.

---

## 2. Мотивация

### 2.1 Почему нужно кеширование

Кеширование предлагается внедрить, чтобы:

- ускорить загрузку первой страницы MES (дашборд заказов по статусам);
- уменьшить количество одинаковых и часто повторяющихся чтений из БД MES;
- снизить нагрузку на MES DB (CPU/IO/коннекты) при линейном росте заказов;
- улучшить пользовательский опыт операторов (быстрее увидеть новые заказы);
- повысить устойчивость при всплесках нагрузки (партнёры / сезонность).

### 2.2 Какие проблемы решаем

- **Высокая latency** для операции чтения списка заказов по статусам.
- **Рост нагрузки на БД** из-за повторяющихся запросов (дашборд открывают часто).
- **Пиковые нагрузки** приводят к деградации времени ответа и UX.

### 2.3 Что именно кешируем

Кешируем **результаты чтения** для наиболее частых запросов в MES:

1. Списки заказов по статусам для дашборда:

- `NEW/INITIATED/...` (в рамках MES — статусы, которые отображаются на первой странице)
- `MANUFACTURING_STARTED`
- `PACKAGING`
- `SHIPPED`
- и т.д.

2. (Опционально) агрегаты для виджетов дашборда:

- количество заказов по статусам (counts).

---

## 3. Предлагаемое решение

### 3.1 Тип кеширования: серверное

Выбираем **серверное кеширование** (на стороне backend MES), потому что:

- данные динамические и общие для множества пользователей (операторов);
- важна скорость именно API MES;
- клиентское кеширование (в браузере) не снижает нагрузку на сервер и БД при большом количестве операторов;
- нужна единая точка контроля TTL/инвалидации и защиты от устаревания данных.

### 3.2 Технология кеша

- Выделенный распределённый кеш: **Redis** (Managed Redis / Redis Cluster при необходимости).
- Подключение из MES backend (C#) через клиент (например, StackExchange.Redis).

Почему Redis:

- низкая задержка чтения/записи;
- поддержка TTL;
- возможность атомарных операций;
- масштабирование независимо от MES API и БД.

### 3.3 Паттерн кеширования: Cache-Aside

Выбираем **Cache-Aside**, потому что:

- чтение (дашборд) происходит часто;
- запись/изменение статуса — существенно реже чтения;
- Cache-Aside проще внедрять поэтапно и безопасно (без перестройки всех write-потоков);
- позволяет деградировать к БД при проблемах с кешем.

Почему хуже другие:

- **Write-Through**: усложняет write-path и требует жёсткой синхронизации кеша при каждой записи; риск замедления операций изменения статуса.
- **Refresh-Ahead**: требует прогнозирования ключей/нагрузки и фоновых задач прогрева; сложно для множества фильтров/страниц/статусов, при этом “самые новые заказы” могут всё равно быть не прогреты.

### 3.4 Ключи кеша и гранулярность

#### Кешируемые сущности

1. Список заказов (по статусу + пагинация):

- ключ: `mes:orders:list:{status}:{page}:{pageSize}`
- value: сериализованный список заказов (JSON/MsgPack)

2. (Опционально) счётчики по статусам:

- ключ: `mes:orders:count:{status}`
- value: integer

#### TTL

- для списков: **5–15 секунд** (короткий TTL, чтобы новые заказы появлялись быстро)
- для счётчиков: **5–30 секунд**

TTL подбирается экспериментально по метрикам и обратной связи операторов (компромисс между свежестью и выигрышем в нагрузке).

## 5. Стратегия инвалидации кеша

### 5.1 Выбранная стратегия: комбинированная (по ключу + временная)

Выбираем **комбинированную стратегию**:

1. **Временная (TTL)** — как базовый механизм защиты от “вечного” устаревания, а также чтобы не поддерживать огромное количество ключей бесконечно.
2. **Инвалидация по ключу (программная)** при изменении статуса заказа — чтобы минимизировать период устаревших данных и ускорить появление новых заказов на дашборде.

Это особенно важно, потому что операторы заинтересованы видеть самые новые заказы максимально быстро.

### 5.2 Как именно инвалидируем

При изменении статуса заказа (например, `MANUFACTURING_STARTED`):

- Обновляем запись в БД (источник истины).
- Инвалидируем ключи:
  - списки для “старого” статуса (страницы 1..N — можно ограничить первыми страницами)
  - списки для “нового” статуса (страницы 1..N)
  - счётчики по статусам (если используются)

**Важно:** не обязательно удалять “все страницы”. Операторам критичны первые страницы с новыми заказами, поэтому можно инвалидировать только:

- `page = 1..3` (или 1..5) для затронутых статусов,
- остальные страницы пусть протухают по TTL.

### 5.3 Почему эта стратегия подходит

- TTL гарантирует, что даже если где-то забыли инвалидировать ключ — данные обновятся автоматически через короткое время.
- Инвалидация по ключу обеспечивает “почти актуальность” там, где это важно (дашборд операторов).
- Компромисс по сложности: не требуется сложная система событийного пересчёта всех ключей.

## 6. Сравнительный анализ стратегий инвалидации

| Стратегия                           | Плюсы                                                                                | Минусы                                                                                               | Почему (не) подходит MES                          |
| ----------------------------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| Только TTL                          | Очень просто, минимум кода                                                           | Данные могут быть устаревшими до истечения TTL; “новые” заказы появятся с задержкой                  | Недостаточно: операторам важны самые новые заказы |
| Только по ключу                     | Максимальная актуальность                                                            | Сложно: нужно точно понимать, какие ключи затронуты; риск “забыть” ключ и получить неконсистентность | Можно, но рискованно без TTL-подстраховки         |
| Программная (событийная) пересборка | Можно поддерживать горячий кеш всегда актуальным                                     | Сложнее, дороже в реализации; нужны фоновые воркеры/очереди; риск гонок                              | Избыточно для старта, можно развивать позже       |
| Комбинированная (TTL + по ключу)    | Баланс простоты и актуальности; TTL страхует ошибки, инвалидация ускоряет обновление | Чуть сложнее, чем только TTL                                                                         | Лучший стартовый вариант для MES                  |

## 7. Дополнительное задание: несколько решений (варианты)

Ниже приведены два возможных решения. Оба решают проблему latency, но отличаются сложностью и качеством актуальности данных.

### 7.1 Вариант 1 — Redis + Cache-Aside + комбинированная инвалидация (рекомендуемый)

- Redis как общий распределённый кеш.
- Cache-Aside для чтения списков и счётчиков.
- TTL 5–15 секунд + инвалидация первых страниц по статусам при изменениях.

### 7.2 Вариант 2 — In-memory кеш на инстансе MES API + TTL

- Использовать MemoryCache (внутри .NET) без Redis.
- TTL 3–10 секунд.
- Без общей консистентности между инстансами (если появится масштабирование).

---

## 8. Сопоставление решений

| Критерий                 | Решение 1: Redis + Cache-Aside | Решение 2: In-memory + TTL       |
| ------------------------ | ------------------------------ | -------------------------------- |
| Ускорение дашборда       | Высокое                        | Среднее/высокое                  |
| Снижение нагрузки на БД  | Высокое                        | Среднее                          |
| Масштабирование          | Хорошее (общий кеш)            | Плохое (кеш локальный)           |
| Консистентность          | Выше (единый кеш)              | Ниже (разные кеши на инстансах)  |
| Сложность внедрения      | Средняя                        | Низкая                           |
| Стоимость инфраструктуры | Выше                           | Ниже                             |
| Риски при росте нагрузки | Низкие/контролируемые          | Высокие при добавлении инстансов |

---

## 9. Заключение

Рекомендуется **Решение 1 (Redis + Cache-Aside + комбинированная инвалидация)**, потому что оно:

- лучше масштабируется при росте заказов и при потенциальном увеличении числа инстансов MES API;
- сильнее снижает нагрузку на MES DB;
- даёт баланс между актуальностью данных и простотой реализации за счёт TTL + selective invalidation;
- позволяет развивать решение дальше (например, добавить refresh-ahead для самых горячих ключей или аналитические счётчики).

Решение 2 подходит как быстрый временный шаг, но станет ограничением при росте нагрузки и масштабировании приложения.
